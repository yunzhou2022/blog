# 项目结构

controller -> facade -> service -> gateway

gateway 包含command， 每个command 都是对一个外部服务的调用，支持超时、异步、降级等行为。

# 新增一个gateway，怎么组织代码

gateway 封装外部调用， command 继承 thriftCommand，封装一个具体的 Thrift 方法调用

# gateway 与 facade 的区别

facade 是对内部service的组合封装， gateway 调用的是外部服务

# 打开首页， app发起请求，数据是怎么流转的

controller -》 facade -》 service -》 gateway -》 外部微服务（thrift、http）

# graph、node 模型， 隶属于service

- graph 

聚合node，并发请求接口

- node

有request 和response, 有 remoteQueryNode 和 localQueryNode 两种，

remoteQueryNode 负责请求数据， localQueryNode 负责组装数据

# 如果批量发起的异步任务有一个失败或者超时了，怎么处理

跳过这个失败的任务， 返回其他 zone， 保证局部降级，整体可用

# 并行的目的

不相互等待，设置超时时间， 统一获取结果。

# 缓存null值的作用

防止缓存击穿

# 设计缓存的注意事项

1. null值也缓存， 防止缓存击穿
2. 注意key 的设计，防止出现重复key， 也防止key太细不能命中缓存


# 3级缓存

1， 内存缓存

纳秒级，只在单个jvm中生效，重启丢失，多个实例不共享

当qps 极高的时候，内存缓存能挡住大部分热点请求，只有miss时才fallback到redis，如果每次都请求redis， 那么redis会成为瓶颈。

2， redis 分布式缓存

毫秒级往返，跨实例共享，容量大，持久化

3， 分布式计数器

tair 针对 计数，排行等场景优化，原子性更强，在极高并发的场景下比redis 更稳定，支持按照时间窗口自动过期。

# 监控与日志

除了正常log打印日志外， 还需要tracer透传traceId， 追踪跨线程、跨服务的日志。

# log注意点

不要把list完整tostring打印， 最好是打印length和前几个
