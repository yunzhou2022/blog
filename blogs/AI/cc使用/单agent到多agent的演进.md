# 为什么需要架构式思维

- 能够快速拆解一个新项目，从高纬度看项目。

# 什么时候升级到多Agent

- 上下文管理挑战
- 分布式开发需求

# 四种核心多agent设计模式

1. sub-agents

- 主管拆解任务，委派多个专业sub-agent并行执行
- 上下文隔离，无状态 sub-agent、用户仅与主管交互
- 并行效率高，防止信息串扰
- 代价： token 成本增加
- 场景： 跨领域研究，个人助手协调多系统

2. skills

- 架构： 单agent通过Skill.md按需加载能力，共享上下文
- 特点： 对话连续，轻量prompt切换，上下文共享，交互路径最短
- 优势：响应延迟低、体验流畅、适合能力种类多，但是单次调用少的场景
- 劣势：上下文累积，隔离性弱
- 区别：sub-agents是独立上下文， skills是共享上下文

3. 接力棒模式（handoffs）

- 架构： agent 根据阶段状态显示调用handoff工具移交控制权
- 实现方式： 通过prompt+状态约束+工程结构模拟
- 三要素：明确的阶段状态、角色约束的agent视角、显示完成状态
- 示例： 客服流程（信息收集-》诊断-》解决），每阶段输出“进入下一阶段”触发转换
- 优势： 流程完整、体验自然；
- 劣势： 严格顺序执行，无法并行

4. router

- 架构： Router 对用户请求语义拆分，并行发给专业 agent，最后合成结果
- 形态： 可作为 tool 或 轻量 sub-agent实现
- 示例： 用户问“退货政策与销售数据”，Router分发到政策 Agent 和数据 Agent
- 优势： 职责清晰、并行能力强、上下文隔离
- 劣势： 无状态，难复用历史对话。
- 场景： 垮数据源查询（企业知识库）

# router与sub-agent的区别

router 只做简单的分发，适合处理简单任务

sub-agent模式中， 主管需要收集sub-agent的产出， 然后总结
