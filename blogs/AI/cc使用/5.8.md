# Superpowers 分析

> 仓库：https://github.com/obra/superpowers  
> 作者：Jesse Vincent（Prime Radiant）  
> 当前版本：v5.1.0

Superpowers 不是一个工具，而是一套**针对 AI 编码代理的软件工程方法论**，以可安装插件的形式分发。核心思路是：通过强制 AI 在行动前调用对应的"技能"，把好的工程实践内化为代理的默认行为。

支持 Claude Code、Cursor、Copilot CLI、Codex、Gemini CLI 等多个平台。

---

## 核心设计：技能必须调用

Superpowers 的关键机制是 `using-superpowers` 这个元技能，它在每次对话开始时加载，强制 AI 遵循一条规则：

> 如果某个技能哪怕有 1% 的可能适用，就必须调用它。

文件里专门列了一张"红旗"表，枚举了 AI 可能用来绕过这条规则的各种理由：

| AI 的想法 | 现实 |
|-----------|------|
| "这只是个简单问题" | 问题也是任务，检查技能 |
| "我需要先了解更多背景" | 技能检查在澄清问题之前 |
| "我记得这个技能怎么用" | 技能会演进，读最新版本 |
| "这个技能太重了" | 简单的事会变复杂，用它 |

这个设计是反直觉的——AI 天然倾向于"我知道怎么做，直接做"，而 Superpowers 强迫它先停下来走流程。

## 14 个技能：覆盖开发全生命周期

### 需求与规划

- **`brainstorming`**：任何功能开发前必须调用。先探索用户意图、需求和设计，再进入实现。防止 AI 一拿到需求就开始写代码。
- **`writing-plans`**：有规格说明或多步骤需求时，先写实现计划，再动代码。
- **`executing-plans`**：在独立会话中按计划执行，带评审检查点。

### 实现

- **`test-driven-development`**：任何功能或修复前调用。强制执行红绿重构循环。
- **`subagent-driven-development`**：当前会话中执行有多个独立任务的计划时，用子代理并行处理。
- **`dispatching-parallel-agents`**：面对 2 个以上互相独立的任务时，并行分发代理。
- **`using-git-worktrees`**：需要工作区隔离时调用。优先使用平台原生的 worktree 工具，其次回退到 `git worktree`。

### 验证

- **`verification-before-completion`**：在声称"完成"、"修好了"、"测试通过"之前调用。必须先跑验证命令、确认输出，再做任何成功断言。核心原则：**证据先于结论**。
- **`systematic-debugging`**：遇到任何 bug、测试失败或意外行为时，在提出修复方案前调用。

### 代码评审

- **`requesting-code-review`**：完成任务或合并前调用。分发子代理做独立评审。
- **`receiving-code-review`**：收到评审反馈后，在实施建议前调用。要求技术严谨性，而非盲目同意。

### 收尾

- **`finishing-a-development-branch`**：实现完成、测试通过后调用。提供合并、PR 或清理的结构化选项。

### 元技能

- **`using-superpowers`**：每次对话开始时的引导技能，建立整套技能调用规则。
- **`writing-skills`**：创建或编辑技能时调用，确保技能质量。

## 与 Cursor Team Kit 的对比

两者都是面向开发工作流的技能插件，但定位不同：

| | Superpowers | Cursor Team Kit |
|--|--|--|
| **定位** | 工程方法论（如何做） | 工程工具集（做什么） |
| **强制性** | 强制调用，有元技能约束 | 按需触发 |
| **覆盖范围** | 全开发生命周期 | CI/PR/代码质量 |
| **核心价值** | 防止 AI 跳过工程流程 | 自动化重复性工程操作 |
| **平台支持** | 多平台（CC、Cursor、Codex 等） | 主要面向 Cursor |

两者可以互补使用：Superpowers 管流程纪律，Cursor Team Kit 管具体操作。

## 值得关注的细节

**`verification-before-completion` 的设计哲学**：这个技能的存在承认了一个现实——AI 有很强的倾向去声称"完成了"而不做实际验证。强制在提交或 PR 前跑验证命令，是对这种行为模式的系统性纠正。

**v5.1.0 对贡献者的警示**：Release Notes 里提到，过去 100 个关闭的 PR 中有 94% 被拒绝，原因是"AI 生成的低质量内容"——没读 PR 模板、开重复 issue、伪造问题描述。这个数字本身就说明了为什么需要 Superpowers 这类方法论约束。
